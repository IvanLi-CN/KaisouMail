# Cloudflare Token 权限

这一页只回答 4 个问题：

1. 正式环境应该怎么配
2. 两把 token 分别放哪里
3. 每把 token 需要哪些权限
4. 什么时候才允许偷懒共用一把 token

## 结论

**正式环境默认推荐拆成两把 token。**

- Worker 运行时用一把 `runtime token`
- GitHub Actions 部署用一把 `deploy token`
- 两边的变量名都继续叫 `CLOUDFLARE_API_TOKEN`
- 区别只在于：**存放位置不同，值也不同**

## 正式环境推荐接法

| 用途 | 存放位置 | 密钥名 | 应填什么 |
| --- | --- | --- | --- |
| 运行时域名管理 | Cloudflare `cf-mail-api` Worker secret | `CLOUDFLARE_API_TOKEN` | runtime token |
| 部署流水线 | GitHub Actions repository secret | `CLOUDFLARE_API_TOKEN` | deploy token |

照着这个关系配置即可，不需要修改代码里的变量名。

## Runtime token

### 放哪里

- Cloudflare Dashboard
- Worker `cf-mail-api`
- Secrets
- `CLOUDFLARE_API_TOKEN=<runtime token>`

### 最小权限

当前运行时代码路径会调用：

- `GET /zones`
- `GET /zones/:zone_id`
- `POST /zones/:zone_id/email/routing/enable`
- `POST /zones/:zone_id/email/routing/dns`
- `POST /zones/:zone_id/email/routing/rules`
- `DELETE /zones/:zone_id/email/routing/rules/:rule_id`

所以 runtime token 只需要：

| 权限 | 必须 | 用途 |
| --- | --- | --- |
| `Zone: Zone: Read` | 是 | 列出和校验 zone |
| `Zone: Email Routing Rules: Edit` | 是 | 创建/删除邮箱 routing rule |
| `Zone: Zone Settings: Edit` | 是 | 启用 Email Routing |

### Scope

- 必须覆盖所有要接入 CF Mail 的 zones
- 如果只覆盖 `707979.xyz`，那别的域名即使在 Cloudflare 里存在，也不能在项目里正常启用

## Deploy token

### 放哪里

- GitHub repository secrets
- `CLOUDFLARE_API_TOKEN=<deploy token>`

### 最小权限

当前部署流程会做：

- `wrangler deploy` 发布 Worker
- `wrangler pages deploy` 发布控制台
- 远程 D1 migration

所以 deploy token 需要：

| 权限 | 必须 | 用途 |
| --- | --- | --- |
| `Account: D1: Edit` | 是 | 远程 D1 migration |
| `Account: Workers Scripts: Edit` | 是 | 发布 Worker |
| `Account: Cloudflare Pages: Edit` | 是 | 发布 Pages |
| `Zone: Workers Routes: Edit` | 是 | 绑定 Worker 自定义域路由 |

### Scope

- Account scope：目标 Cloudflare account
- Zone scope：所有会被 Worker route / custom domain 使用到的 zones

## 快速试用：共用一把 token

只在这些情况下允许共用：

- 单人维护
- 自建环境
- 临时验证
- 低风险内部演示

这时你可以把同一把 token 同时填到：

- GitHub Actions secret `CLOUDFLARE_API_TOKEN`
- `cf-mail-api` Worker secret `CLOUDFLARE_API_TOKEN`

这把共享 token 必须满足并集权限：

- `Zone: Zone: Read`
- `Zone: Email Routing Rules: Edit`
- `Zone: Zone Settings: Edit`
- `Account: D1: Edit`
- `Account: Workers Scripts: Edit`
- `Account: Cloudflare Pages: Edit`
- `Zone: Workers Routes: Edit`

## 为什么正式环境不建议共用

- 它把部署权限和线上运行时变更权限绑在了同一把凭据上
- GitHub Actions secret 或 Worker secret 任一侧泄露，影响面都会变大
- Cloudflare 官方也更倾向于让长期自动化使用独立、边界清晰的 token，并强调最小权限和资源范围约束

所以这里的默认决策不是“由你自己权衡”，而是：

- **正式环境：拆分**
- **快速试用：可共用**

## 最小排障顺序

1. 确认 token 资源范围覆盖目标 zone
2. 确认 runtime token 放在 `cf-mail-api` Worker secret，而 deploy token 放在 GitHub Actions secret
2. 确认 token 具备 `Zone Settings: Edit`
3. 确认 `EMAIL_ROUTING_MANAGEMENT_ENABLED=true`
4. 确认 `EMAIL_WORKER_NAME` 已配置
5. 回到控制台域名页执行“重试接入”

## English summary

Production should use two different token values under the same secret name `CLOUDFLARE_API_TOKEN`: one stored as the `cf-mail-api` Worker secret for runtime, and one stored as the GitHub Actions repository secret for deploy. Runtime needs `Zone: Zone: Read`, `Zone: Email Routing Rules: Edit`, and `Zone: Zone Settings: Edit`. Deploy needs `Account: D1: Edit`, `Account: Workers Scripts: Edit`, `Account: Cloudflare Pages: Edit`, and `Zone: Workers Routes: Edit`. A single shared token is kept only for quickstart and internal low-risk use.
